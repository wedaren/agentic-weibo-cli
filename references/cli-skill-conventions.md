# CLI + Skill Conventions

这份文档定义当前仓库作为“完整 CLI + skill 整合项目”时遵循的约定。

## 目标

- 同一套能力既能被终端用户直接使用，也能被 agent 稳定调用
- CLI 负责可执行、可脚本化、可诊断
- skill 负责什么时候调用、先后顺序、交互约束与副作用控制
- 评估、文档、自检围绕同一套命令入口演进，避免 CLI 和 skill 脱节

## CLI 通用规范

### 发现性

- 必须支持 `--help`
- 必须支持 `--version`
- 子命令命名保持短、小写、动宾明确，例如 `login`、`list`、`reposts`
- 参数名使用长选项形式，例如 `--weibo-id`、`--browser-user-data-dir`

### 输出与退出码

- 正常结果写到 stdout
- 错误写到 stderr
- 成功退出码为 `0`
- 失败退出码为非 `0`
- 默认输出是便于人读的文本
- 统一支持 `--json` 作为机器可读输出契约

当前退出码约定：

- `0`：成功
- `2`：参数或命令用法错误
- `3`：鉴权失败或登录态无效
- `4`：业务接口或运行时操作失败
- `5`：网络请求失败
- `10`：未分类内部错误

### 交互约定

- 默认优先非交互，便于 agent 与脚本直接调用
- 只有显式要求时才进入交互模式，例如 `login --prompt`
- 会改变外部状态的动作不由 CLI 自行确认，而由上层 skill / agent 先确认后调用

### 配置与状态

- 配置优先级固定为：环境变量 > 本地文件
- 本地状态写入 `.local/`，不提交到仓库
- 对鉴权失败返回明确错误，不伪装成空结果

### 自动化友好性

- 机器输出字段应尽量稳定、可预期
- 自检脚本应覆盖帮助、关键命令和机器输出契约
- 新增命令时，README、SKILL、commands、tests、evals 至少同步其一条可执行示例

## Skill 侧规范

### 何时由 skill 介入

- 用户提出微博登录、查看、评论、点赞、删除等任务时，应先命中 skill
- skill 负责把自然语言任务映射成 CLI 子命令，而不是在对话里复述接口细节

### Skill 与 CLI 的职责边界

- CLI 负责“怎么执行”
- skill 负责“该不该执行、先执行什么、是否需要确认”
- 破坏性操作的确认规则写在 `SKILL.md`，不要分散到 CLI 里做交互式确认

### 机器输出使用时机

- 面向最终用户时默认保留文本输出
- 需要把结果交给脚本、评估器或其他 agent 继续处理时，优先使用 `--json`
- 评估、回归测试和自动化流水线应优先依赖 `--json`

## 项目级最佳实践清单

一个完整的 CLI + skill 整合项目，至少应做到：

- 有清晰的入口脚本，例如 `scripts/weibo-cli`
- 有稳定的 skill 文档入口，例如 `SKILL.md`
- 有面向用户的 README
- 有命令参考文档，例如 `references/commands.md`
- 有架构说明文档，例如 `references/architecture.md`
- 有自检脚本，例如 `scripts/self-check`
- 有评估用例与 workspace 模板，例如 `evals/`
- 有单元测试覆盖关键行为

## 当前仓库已支持

- `--help` / `--version`
- 文本输出与 `--json` 机器输出
- 明确的 stdout / stderr 分离
- 稳定的分类退出码
- 环境变量优先于本地 session 文件
- skill 文档、命令文档、架构文档、评估模板、自检脚本
- 针对 CLI 扩展与会话行为的单元测试

## 后续可继续补的规范

- Shell completion
- `--output text|json` 这样的显式输出模式别名
- 面向 CI 的 JSON schema 校验
